# SML-TL: Stateless Optical Transport

### The "Radio-Broadcast" Time-Locked Transfer Protocol (v2)

## Abstract

**SML-TL v2** (Secret Memory Locker - Time Locked, version 2) is a highly experimental, hybrid protocol designed for the secure, stateless transfer of high-entropy data between an air-gapped PC and a mobile device. It eliminates shared clipboards, cloud vaults, and persistent network connections by utilizing a **Zero-Parameter Key Faucet** and a **Dual-Entropy Reconstruction** model.

## 1. The Core Problem

Traditional secret sharing relies on "Store-and-Forward" logic:

1. Data is encrypted.
2. Encrypted payload is stored on a centralized server.
3. Recipient fetches it via a unique link.

**SML-TL v2** completely removes the "Store" component. The server never receives, sees, or processes the encrypted payload. Instead, it acts as a cryptographic "radio station," constantly broadcasting volatile time-keys required for local reconstruction.

## 2. Technical Specification

### A. Key Components

* **Payload:** The generated secret (e.g., password + resource name).
* **Nonce (N):** A unique 8-character random string generated locally on the source device.
* **Time Buckets (B_now, B_next):** Deterministic time-based integers calculating the current and next 2-minute windows:
`B = floor(CurrentTimestamp / 120000)`
* **Server Entropy (k_now, k_next):** A dual-array of 256-bit hashes generated by a stateless Cloudflare Worker:
`k = HMAC_SHA256(SecretSalt, B)`
* **Reconstruction Key (K_final):** The final AES-GCM key derived on the client side.

### B. The Encryption Flow (Source Device - PC)

1. Generate the Payload using the Memory Echo engine.
2. Calculate the current time bucket **B** locally to derive the encryption key.
3. Combine the server-aligned entropy (**k1**) with the local nonce (**N**) to form the master key material.
4. Encrypt the Payload using AES-GCM (256-bit).
5. Encode the result into an ultra-compact, parameterless QR code payload (CSV-style):
`[https://secretmemorylocker.com/tl#](https://secretmemorylocker.com/tl#){nonce},{k_cipher},{r_cipher}`

### C. The Decryption Flow (Target Device - Mobile)

1. **Optical Transport:** The mobile camera scans the QR code and parses the CSV-style hash: `nonce`, `k_cipher`, and `r_cipher`.
2. **Zero-Parameter Fetch:** The device makes a GET request to the stateless Worker. **No parameters are sent.** The Worker replies with a JSON array of keys for the current and next time windows: `[kNow, kNext]`.
3. **Temporal Self-Alignment:** The client-side JS attempts to decrypt the payload using **kNow**. If it fails (due to a time-slot transition), it automatically attempts decryption with **kNext**.
`K_final = SHA256(k + Nonce + "SML-TL-v2")`
4. **Volatile Decryption:** Upon success, the secret is displayed in the browser's volatile memory, and a 120-second session timer starts before the data is wiped from the DOM.

## 3. Security Model (Why it works)

### 1. True Stateless Infrastructure (The "Key Faucet")

The Cloudflare Worker has no database and accepts no inputs. It only knows its internal `SecretSalt` and the `CurrentTime`. Because the API accepts zero parameters, attackers cannot request historical keys or manipulate the entropy source.

### 2. Dual-Entropy Air-Gap

An attacker needs **both** pieces to compromise the secret:

* **Piece A (The Optical QR):** Contains the encrypted ciphertext and the nonce.
* **Piece B (The Server Broadcast):** Provides the temporary **k1**.
Without the optical scan, the server's broadcast is useless noise. Without the server's time-key, the QR is mathematically unbreakable.

### 3. Native Time-Locking (Absolute Expiration)

Since the URL contains no bucket or time data, the QR code's lifespan is strictly governed by the Worker's rotation. Once the current and next buckets move forward, the keys required for a specific QR code vanish forever. A QR code permanently "dies" after a maximum of **4 minutes**.

## 5. Implementation Status: Production-Ready (v2)

This protocol is the default secure-handoff mechanism for the **Secret Memory Locker (SML)** suite. It is designed for high-security environments where moving a secret from an untrusted PC to a personal mobile device must be completely invisible to network sniffers, proxy logs, and clipboard managers.
