#  SML-TL: Secure Time-Locked Transport Layer

## Abstract
**SML-TL v2** (Secret Memory Locker - Time Locked, version 2) is an experimental hybrid protocol designed for the secure, stateless transfer of high-entropy data between two air-gapped devices. It eliminates the need for shared clipboards, cloud-based vaults, or persistent network connections, utilizing a **Dual-Entropy Reconstruction** model.

## 1. The Core Problem
Traditional secret sharing (e.g., Bitwarden Send, 1Password Psst!) relies on "Store-and-Forward" logic:
1. Data is encrypted.
2. Encrypted payload is stored on a server.
3. Recipient fetches it via a link.

**SML-TL v2** removes the "Store" part. The server never sees the encrypted payload; it only provides a volatile "Time-Key" required for reconstruction.

## 2. Technical Specification

### A. Key Components
*   **$P$ (Payload):** The generated secret (e.g., password + resource name).
*   **$N$ (Nonce):** A unique 8-character random string generated on the source device.
*   **$B$ (Time Bucket):** A deterministic time-based integer, calculated as:
    $$B = \lfloor \frac{CurrentTimestamp}{120000} \rfloor$$ (where 120,000ms = 2 minutes).
*   **$k1$ (Server Entropy):** A dynamic 256-bit hash generated by a stateless Cloudflare Worker:
    $$k1 = HMAC\_SHA256(SecretSalt, B)$$
*   **$K_{final}$ (Reconstruction Key):** The final AES-GCM key.

### B. The Encryption Flow (Source Device - Offline)
1.  Generate **$P$** using the Memory Echo engine.
2.  Receive/Calculate the current bucket **$B$**.
3.  The source device derives the encryption key by combining the server's public entropy $k1$ with the local nonce $N$.
4.  Encrypt **$P$** using AES-GCM (256-bit).
5.  Encode the result into a QR code containing:
    `#SML-TL2|b=BUCKET|n=NONCE|k=CIPHERTEXT_PASS|r=CIPHERTEXT_RES`

### C. The Decryption Flow (Target Device - Mobile)
1.  **Scan:** Camera captures the QR and parses the hash parameters.
2.  **Fetch:** The device makes a GET request to the stateless Worker to retrieve the $k1$ for the specific bucket $B$ mentioned in the QR.
3.  **Derive:** The client-side JS reconstructs $K_{final}$ using the same logic:
    $$K_{final} = SHA256(k1 \parallel N \parallel "SML-TL-v2")$$
4.  **Decrypt:** The payload is decrypted in the browser's volatile memory.

## 3. Security Model (Why it works)

### 1. Zero-Knowledge Infrastructure
The Cloudflare Worker is **stateless**. It doesn't have a database. It only knows its own `SecretSalt` and the `CurrentTime`. It never receives the `Nonce` or the `Ciphertext`.

### 2. The "Two-Piece Puzzle" (Dual Entropy)
An attacker needs **both** pieces to compromise the secret:
*   **Piece A (The QR):** Contains the encrypted data and the nonce.
*   **Piece B (The Server):** Provides $k1$.
Without the QR, the server is useless. Without the server's time-key, the QR is a block of unbreakable high-entropy noise.

### 3. Native Time-Locking (Self-Destruction)
Since $k1$ is tied to the bucket $B$, once the 2-minute window passes, the server will no longer provide the specific $k1$ needed for that QR code (unless the bucket is still within a very small grace period). The QR code effectively "expires" in real-time.

## 4. Comparison with Existing Methods

| Feature | TOTP (Google Auth) | Bitwarden Send | **SML-TL v2** |
| :--- | :--- | :--- | :--- |
| **Storage** | Secret seed is stored | Ciphertext is stored | **Nothing is stored** |
| **Medium** | Manual typing (6 digits) | Network link | **Optical (QR Scan)** |
| **Payload** | Only 6 digits | Large files/text | **Any text secret** |
| **Offline** | Yes | No | **Hybrid (Air-Gapped)** |

## 5. Implementation Status: Experimental
This protocol is currently implemented as part of the **Secret Memory Locker (SML)** suite. It is designed for high-security environments where the user wants to move secrets from a secure PC to a mobile device without trusting the underlying network or clipboard sync services.
